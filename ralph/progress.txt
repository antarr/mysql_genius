## Codebase Patterns
- AI classes follow two constructor signatures: `(client, config, connection)` for server-level features (SchemaReview, VariableReviewer) and `(connection, client, config)` for query-level features (Suggestion, Optimization)
- Desktop FakeAdapter `stub_query` stubs must cover all SHOW/performance_schema queries that downstream service classes issue — add them in the spec's `before` block
- New AI buttons in tab partials use a shared result panel pattern: `{prefix}-ai-result`, `{prefix}-ai-title`, `{prefix}-ai-content`, `{prefix}-ai-close`
- JS ROUTES hash in dashboard.html.erb must be updated whenever new routes are added — buttons reference `ROUTES.route_name`
- FakeAdapter `stub_query` takes a regex pattern, columns array, and rows array-of-arrays
- RuboCop enforces double-quoted strings and literal arrays (not `%w[]`) — use `bundle exec rubocop -A` to auto-correct
- All new AI classes must be required in `gems/mysql_genius-core/lib/mysql_genius/core.rb`
- Specs use `instance_double(MysqlGenius::Core::Ai::Client)` for the AI client mock
- When using `ServerOverview` in an AI class, the FakeAdapter needs stubs for all SHOW GLOBAL STATUS/VARIABLES keys that ServerOverview reads (not just the keys your class cares about)
- `ColumnDefinition.new` requires `primary_key:` keyword — always include it in spec stubs
- RuboCop prefers `to_h { |x| [key, val] }` over `each_with_object({})` for hash construction
